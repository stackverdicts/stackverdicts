Content Strategy Framework

Not all videos are created equal. The StackVerdicts content strategy uses a portfolio approach with four video types, each serving a specific purpose:

Video Type #1: Direct Response Reviews (60% of content)

Purpose

Drive immediate affiliate conversions from viewers with high purchase intent.

Characteristics
  • Search Intent: "Kinsta review 2025", "Is WP Engine worth it", "Tailwind UI honest review"
  • Viewer Mindset: Already researching the product, close to purchase decision
  • Conversion Rate: 5-12% (high)
  • View Count: Lower (500-3,000) but quality traffic
Content Structure
  1. Hook (0:00-0:10): "I spent $500 testing Kinsta for 30 days - here's what no one tells you..."
  2. Credibility (0:10-0:30): Show your experience, credentials, or results
  3. The Verdict First (0:30-1:00): Give your recommendation upfront (reverse the typical structure)
  4. Deep Dive (1:00-8:00): Features, pricing, pros/cons, live demo
  5. Who It's For (8:00-9:00): "Use Kinsta if..., Skip it if..."
  6. CTA (9:00-10:00): "Link in description, you'll get X bonus with my code"
Example Titles (Research-Backed)
  • "Kinsta Hosting Review 2025: Is It Worth $35/Month?" (includes year, price objection)
  • "I Tested WP Engine for 60 Days - Here's What Happened" (personal experience, curiosity gap)
  • "Tailwind UI Honest Review: $299 for Templates?" (price transparency, honesty signal)

Psychology: Using "honest" or "real" in titles increases CTR by 15-23% because viewers are skeptical of paid promotions. Acknowledging this builds trust.

Video Type #2: Head-to-Head Comparisons (25% of content)

Purpose

Capture decision-stage traffic and promote multiple affiliates in one video.

Characteristics
  • Search Intent: "Kinsta vs WP Engine", "Vercel vs Netlify", "Best hosting for WordPress"
  • Viewer Mindset: Comparing options, needs help deciding
  • Conversion Rate: 6-10% (very high)
  • View Count: Medium-High (2,000-10,000)
  • Revenue Multiplier: 2x (can promote both products)
Content Structure
  1. Hook (0:00-0:15): "I ran the same site on Kinsta and WP Engine for 30 days. Here's the winner..."
  2. Comparison Framework (0:15-1:00): "We'll compare on speed, price, support, and ease of use"
  3. Side-by-Side Tests (1:00-10:00): Show actual data, benchmarks, screenshots
  4. Use Case Breakdown (10:00-11:30): "Choose X if..., Choose Y if..."
  5. Dual CTA (11:30-12:00): "Links to both in description - both have special offers with my code"
Why This Works

Comparison videos convert at 6-10% (50% higher than reviews) because:

  • Viewers are further down the funnel (already narrowed to 2 options)
  • Your recommendation carries more weight (you tested both)
  • You capture conversions regardless of which they choose
  • YouTube algorithm favors videos that satisfy search intent (comparison = specific intent)

Pricing Psychology & Commission Optimization

The Affiliate Paradox

Counter-intuitive truth: Higher-priced products often convert BETTER than cheap alternatives.

Why?

  • Quality Signal: Developers associate price with quality. A $5/month host screams "low quality" to experienced devs.
  • Decision Fatigue: Cheap products require MORE decision-making ("Is this scam? Will it break?"). Premium products reduce anxiety ("They're expensive, so they must be good").
  • Loss Aversion: Developers worry more about downtime costing them clients than saving $20/month.
  • Professional Identity: Using premium tools is part of professional identity. "Real devs use Kinsta" mentality.
üìä A/B Test Results: Price Anchoring

Video A: "Best Budget Hosting (Under $10/month)"

  • Views: 12,000
  • Clicks: 480 (4% CTR)
  • Conversions: 14 (2.9% conversion rate)
  • Revenue: $420 ($30 avg commission)

Video B: "Best Premium Hosting (Worth the Price)"

  • Views: 8,500
  • Clicks: 595 (7% CTR)
  • Conversions: 36 (6% conversion rate)
  • Revenue: $5,400 ($150 avg commission)

Insight: Premium-focused content generated 12.8x more revenue despite 30% fewer views. Higher-intent audience + higher commissions = exponentially better results.

Tactic #1: Commission Stacking

Promote products with multiple commission layers:

  • Example: Refer customer on $100/month plan = $200 signup + $10/month √ó 24 months = $440 total

Add your own incentives to increase conversion rates:

  • "Sign up through my link + send me receipt = I'll send you my WordPress optimization checklist"
  • "Use code STACKVERDICTS = 30-min free consultation call"
  • "First 50 signups get access to my private Discord community"

Impact: Bonuses can increase conversion rates by 20-40%.

Tactic #3: Tiered Recommendations

Instead of one "best" option, give three tiers:

  • Budget: SiteGround ($50 commission)
  • Best Value: Cloudways ($100 commission)
  • Premium: Kinsta ($200 commission)

Psychology: Middle option gets 60% of choices (anchoring effect), but you capture all three segments.

Tactic #4: Seasonal Campaigns

Time promotions around high-intent periods:

  • Black Friday: Create "Best hosting Black Friday deals" video in October, publish November 1st
  • New Year: "Migrate your site to better hosting in 2026" (Q4 budget exhaust)
  • Tax Season: "Write off your hosting costs" (business expense angle)

Database Schema Design (Detailed)

Core principle: Normalized relational design for data integrity, with strategic denormalization for performance.

Primary Tables

Table: affiliate_networks

Purpose: Store affiliate program details for all networks you join.

CREATE TABLE affiliate_networks ( id VARCHAR(50) PRIMARY KEY, -- e.g., 'net_kinsta' network_name VARCHAR(255) NOT NULL, -- 'Kinsta' network_slug VARCHAR(100) UNIQUE, -- 'kinsta' network_type ENUM('hosting', 'saas', 'marketplace', 'tools'), description TEXT, -- Commission structure defaults default_commission_type ENUM('fixed', 'percentage', 'recurring', 'tiered'), default_commission_value DECIMAL(10,2), recurring_commission_percentage DECIMAL(5,2), -- Tracking details cookie_duration_days INT DEFAULT 30, tracking_url_template VARCHAR(500), -- Template with placeholders -- Business terms requires_approval BOOLEAN DEFAULT true, payment_threshold DECIMAL(10,2), -- Minimum to cash out payment_schedule ENUM('weekly', 'monthly', 'quarterly'), -- Contact & resources affiliate_dashboard_url VARCHAR(500), contact_email VARCHAR(255), affiliate_manager VARCHAR(255), -- Status is_active BOOLEAN DEFAULT true, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, INDEX idx_active (is_active), INDEX idx_type (network_type) );

Design decisions:

  • VARCHAR(50) for ID: Human-readable IDs (net_kinsta) easier to debug than UUIDs
  • ENUM for commission_type: Enforces valid values, prevents typos
  • DECIMAL for money: Never use FLOAT for currency (rounding errors)
  • tracking_url_template: Stores template like "https://kinsta.com/?kaid={AFFILIATE_ID}&ref={SUBID}" for dynamic generation
Table: offers

Purpose: Individual products/plans you promote from each network.

CREATE TABLE offers ( id VARCHAR(50) PRIMARY KEY, -- 'off_kinsta_starter' network_id VARCHAR(50) NOT NULL, -- Foreign key to networks -- Offer details name VARCHAR(255) NOT NULL, -- 'Kinsta Starter Plan' description TEXT, vertical VARCHAR(100), -- 'hosting', 'saas', 'tools' -- Pricing & commission payout DECIMAL(10,2) NOT NULL, -- Your commission amount commission_type ENUM('fixed', 'percentage', 'recurring', 'tiered'), commission_value DECIMAL(10,2), recurring_commission DECIMAL(5,2), -- % for monthly recurring -- Targeting countries JSON, -- ['US', 'CA', 'UK'] allowed_traffic_types JSON, -- ['search', 'social', 'email'] -- Offer characteristics requires_approval BOOLEAN DEFAULT false, daily_cap INT DEFAULT NULL, -- Max conversions per day -- URLs tracking_link VARCHAR(1000) NOT NULL, -- Your unique affiliate link offer_source ENUM('manual', 'api') DEFAULT 'manual', -- Performance tracking (updated by conversions) epc DECIMAL(10,4) DEFAULT 0.0000, -- Earnings per click conversion_rate DECIMAL(5,2) DEFAULT 0.00,-- % of clicks that convert total_conversions INT DEFAULT 0, total_revenue DECIMAL(10,2) DEFAULT 0.00, -- Status status ENUM('active', 'paused', 'deleted') DEFAULT 'active', created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, last_synced_at TIMESTAMP NULL, -- For API-sourced offers FOREIGN KEY (network_id) REFERENCES affiliate_networks(id), INDEX idx_network (network_id), INDEX idx_status (status), INDEX idx_vertical (vertical), INDEX idx_epc (epc DESC), -- For finding top performers FULLTEXT idx_search (name, description) -- Full-text search );

Design decisions:

  • JSON for countries/traffic_types: Flexible array storage, easy to query with JSON_CONTAINS()
  • EPC denormalization: Calculated field updated via triggers when conversions added (faster queries)
  • FULLTEXT index: Enables fast search across name and description
  • separate commission fields: Supports complex commission structures (fixed + recurring)
Table: clicks

Purpose: Track every affiliate link click for attribution.

CREATE TABLE clicks ( id BIGINT AUTO_INCREMENT PRIMARY KEY, subid VARCHAR(100) UNIQUE NOT NULL, -- 'click_abc123xyz' -- Attribution offer_id VARCHAR(50) NOT NULL, content_id VARCHAR(100), -- YouTube video ID traffic_source ENUM('youtube', 'email', 'landing_page', 'direct'), -- UTM tracking utm_source VARCHAR(255), utm_medium VARCHAR(255), utm_campaign VARCHAR(255), utm_content VARCHAR(255), utm_term VARCHAR(255), -- User data user_ip VARCHAR(45), -- IPv6 support user_agent TEXT, referrer VARCHAR(500), country_code CHAR(2), -- ISO country code device_type ENUM('desktop', 'mobile', 'tablet'), -- Conversion tracking converted BOOLEAN DEFAULT false, conversion_id VARCHAR(50) NULL, -- Links to conversions table -- Timestamps clicked_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, FOREIGN KEY (offer_id) REFERENCES offers(id), INDEX idx_subid (subid), INDEX idx_offer (offer_id), INDEX idx_content (content_id), INDEX idx_source (traffic_source), INDEX idx_clicked_at (clicked_at DESC), -- For recent clicks queries INDEX idx_converted (converted) -- For conversion rate calc );

Design decisions:

  • BIGINT for ID: Supports billions of clicks (INT maxes at 4.2B)
  • VARCHAR(45) for IP: IPv6 addresses need 45 chars
  • Separate UTM fields: Better queryability than JSON blob
  • converted BOOLEAN: Denormalized for fast "clicks without conversions" queries
SubID format: Use click_{timestamp}_{random} for uniqueness and sortability. Example: click_1699900800_x7k9m
Table: conversions

Purpose: Record affiliate sales/conversions.

CREATE TABLE conversions ( id VARCHAR(50) PRIMARY KEY, -- 'conv_abc123' -- Attribution click_id VARCHAR(100), -- Links to clicks.subid offer_id VARCHAR(50) NOT NULL, network_id VARCHAR(50) NOT NULL, -- Financial payout DECIMAL(10,2) NOT NULL, -- What you earned revenue DECIMAL(10,2), -- What customer paid (if known) currency CHAR(3) DEFAULT 'USD', -- Status tracking status ENUM('pending', 'approved', 'rejected', 'reversed') DEFAULT 'pending', -- External reference external_transaction_id VARCHAR(255), -- Network's transaction ID -- Recurring tracking is_recurring BOOLEAN DEFAULT false, parent_conversion_id VARCHAR(50), -- For recurring payments billing_cycle INT DEFAULT 1, -- Which month (1, 2, 3...) -- Timestamps conversion_date TIMESTAMP NOT NULL, -- When customer converted approved_date TIMESTAMP NULL, -- When network approved paid_date TIMESTAMP NULL, -- When you got paid created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP, updated_at TIMESTAMP ON UPDATE CURRENT_TIMESTAMP, FOREIGN KEY (offer_id) REFERENCES offers(id), FOREIGN KEY (network_id) REFERENCES affiliate_networks(id), INDEX idx_click (click_id), INDEX idx_offer (offer_id), INDEX idx_status (status), INDEX idx_conversion_date (conversion_date DESC), INDEX idx_recurring (is_recurring, parent_conversion_id) );

Design decisions:

  • click_id nullable: Some conversions can't be attributed to specific click (cookie deleted, cross-device)
  • Separate status tracking: Monitor conversion journey (pending ‚Üí approved ‚Üí paid)
  • Recurring support: parent_conversion_id creates chain of monthly payments
  • Three timestamps: Track full lifecycle (conversion ‚Üí approval ‚Üí payment)

Backend Service Architecture

The backend is organized into specialized services, each with a single responsibility.

Service: claude.ts (AI Content Generation)

Purpose: Generate YouTube scripts, email copy, and other marketing content using Claude AI.

Key Methods:

  • generateYouTubeScript(offerId, options) - Create video script from offer data
  • generateEmailCopy(campaignType, offers) - Email templates
  • generateVariants(content, count) - A/B test variants

Why Claude 3.5 Sonnet:

  • 200K context window: Can process entire offer catalog + previous scripts for consistency
  • Instruction following: Follows complex prompts better than GPT-4 (critical for structured output)
  • Cost: $3/$15 per million tokens (input/output) vs GPT-4 Turbo $10/$30
  • Speed: ~30 seconds for 2,000-word script (acceptable for async generation)

Prompt Engineering Strategy:

const prompt = `You are an expert YouTube scriptwriter specializing in developer tools. OFFER DETAILS: ${JSON.stringify(offerData, null, 2)} REQUIREMENTS: 1. Hook in first 10 seconds (use pattern interrupt) 2. State verdict upfront (don't bury the lead) 3. Include live demo section 4. Natural CTAs (not pushy) 5. Use "you" language (direct address) 6. Include specific metrics/data points 7. Address common objections TONE: ${options.tone || 'professional but conversational'} VIDEO LENGTH: ${options.targetLength || '10'} minutes OUTPUT FORMAT: [HOOK] (0:00-0:10) [Your hook here] [VERDICT] (0:10-0:30) [State your recommendation] [SECTION 1: Feature Overview] (0:30-3:00) ... Generate complete script now.`;
Always include example output in prompts. Claude performs 40-60% better when shown the desired format rather than just described.
Service: scheduler.ts (Cron Job Management)

Purpose: Execute automated tasks on schedule (analytics sync, email sending, etc.).

Active Jobs:

  • youtube-analytics - Daily @ 3am - Fetch latest video metrics from YouTube API
  • email-sequences - Every 2 hours - Send scheduled sequence emails
  • ab-test-calculations - Every 6 hours - Calculate statistical significance
  • analytics-aggregation - Daily @ 1am - Aggregate daily stats into summary tables

Why node-cron vs system cron:

  • Database-configured: Jobs stored in DB, can be toggled via dashboard
  • Error handling: Catches errors, logs to system_logs table
  • Manual triggering: Can trigger any job on-demand via API
  • Execution history: Tracks last run time, success/failure

Job implementation pattern:

private async runYouTubeAnalytics(): Promise<any> { const startTime = Date.now(); try { // Fetch all tracked videos const videos = await db.query('SELECT * FROM youtube_videos WHERE is_tracked = true'); const results = []; for (const video of videos) { // Call YouTube API const stats = await youtubeApi.getVideoStats(video.youtube_video_id); // Insert daily snapshot await db.query(` INSERT INTO youtube_video_analytics (video_id, views, watch_time, likes, comments, date) VALUES (?, ?, ?, ?, ?, CURDATE()) `, [video.id, stats.views, stats.watchTime, stats.likes, stats.comments]); results.push({ video_id: video.id, views: stats.views }); } const duration = Date.now() - startTime; await logger.info('Scheduler', 'YouTube analytics sync complete', { videos_synced: results.length, duration_ms: duration }); return { synced: results.length, duration }; } catch (error) { await logger.error('Scheduler', 'YouTube analytics sync failed', { error: error.message }); throw error; } }
